## 1 项目概述

### 1\.1 项目介绍

本项目是一款**完全本地化部署的 AI 实时互动直播电台桌面应用**，后端采用纯 Python 一体化架构，将电台业务逻辑、AI 推理能力、直播管控能力全部基于 Python 生态实现，与前端 Electron 桌面端无缝衔接。无需公网服务器、无需专业运维知识，普通用户双击即可启动使用，开发者可基于 Python 丰富的生态快速完成二次定制。

### 1\.2 核心功能（1:1 还原原型 \+ 本地增强）

|功能模块|核心能力|对应原型界面元素|
|---|---|---|
|AI 主播实时直播|全自动生成歌曲串词 / 互动口播、拟人化 TTS 语音实时播出、7×24 小时无人值守直播|ON AIR/LIVE 标识、主播口播气泡、REPLAY 重播功能|
|音乐播放与电台管控|本地音乐导入、歌单自动编排、完整播放控制、播放进度同步、音量调节、歌曲收藏|播放控件、进度条、VOL 音量条、QUEUE 队列、FAV 收藏按钮|
|实时互动能力|本地留言输入、点歌指令发送、AI 主播实时响应并融入口播、互动历史回溯|底部消息输入框、用户指令实时响应、口播历史展示|
|沉浸式界面体验|暗黑风赛博质感 UI、实时时间 / 日期显示、音频频谱动效、界面轻量化隐藏、全屏模式|顶部时间日期、频谱动效、HIDE 按钮、本地用户体系|
|本地私有化增强|本地音乐全量支持、AI 模型完全本地化运行、数据本地存储、无网络也可使用、零隐私泄露风险|本地歌单管理、离线模式开关、本地用户数据管理|

1. 个人本地使用：通勤、学习、睡前沉浸式电台陪伴

## 2 普通用户一键启动指南（零代码门槛）

### 2\.1 运行环境要求

|配置档位|最低配置要求|推荐配置要求|适配模式|
|---|---|---|---|
|低配档|Windows 10/MacOS 11，4G 内存，CPU 4 核|Windows 11/MacOS 13，8G 内存，CPU 6 核|云 API 模式（无需本地跑 AI 模型）|
|中配档|Windows 10/MacOS 12，8G 内存，CPU 6 核|Windows 11/MacOS 14，16G 内存，CPU 8 核|本地轻量化 AI 模式|
|高配档|16G 内存，4G 以上显存独立显卡|32G 内存，8G 以上显存独立显卡|本地高质量 AI 模式|

### 2\.2 Windows 一键启动步骤

1. 下载安装包：获取「Butterfly Radio Windows 一键安装包\.exe」（绿色免安装版获取「Butterfly Radio Windows 绿色版\.zip」），安装包已内置完整 Python 运行环境、AI 模型、音频处理组件，无需额外安装任何依赖

2. 安装流程：双击安装包，全程点击「下一步」即可完成安装，无需修改任何配置

3. 启动使用：安装完成后，桌面生成「Butterfly Radio」快捷方式，双击打开软件

4. 自动初始化：软件启动后会自动拉起 Python 后端服务、加载 AI 模型、初始化电台环境，30 秒内即可进入主界面，自动开始 AI 电台直播

5. 模式切换（可选）：

    - 低配电脑：打开右上角「设置」，切换为「云 API 模式」，按提示填入对应 API 密钥，点击保存即可使用，无需本地加载 AI 模型

    - 高配电脑：切换为「本地高质量模式」，按提示一键下载高质量 AI 模型包，重启软件即可生效

### 2\.3 MacOS 一键启动步骤

1. 下载安装包：获取「Butterfly Radio Mac 安装包\.dmg」

2. 安装流程：双击 dmg 文件，将「Butterfly Radio」拖拽到「应用程序」文件夹完成安装

3. 权限放行：首次打开时，若系统提示「无法打开开发者未验证的应用」，打开「系统设置 \- 隐私与安全性」，下滑点击「仍要打开」即可

4. 启动使用：打开应用程序中的「Butterfly Radio」，软件自动完成初始化，进入主界面即可使用，后续操作与 Windows 版本完全一致

### 2\.4 基础使用教程

1. 核心收听：软件启动后自动开始 AI 电台直播，无需任何操作即可沉浸式收听

2. 互动操作：在底部输入框输入留言 / 点歌指令，点击发送，AI 主播会在歌曲间隙实时响应并融入口播

3. 播放控制：可通过界面控件实现暂停 / 播放、切歌、收藏、音量调节、重播主播口播等操作

4. 本地音乐导入：点击左侧「我的歌单」，选择「导入本地音乐」，可将电脑中的音频文件导入电台，AI 会自动编排进直播队列并生成专属口播

5. 个性化设置：点击右上角头像，可修改 AI 主播人设、音色、话术风格，调整界面主题、播放规则等

## 3 全 Python 技术架构与本地适配选型

### 3\.1 整体架构（全 Python 一体化专属设计）

采用**Electron 桌面端 \+ 全 Python 后端一体化架构**，所有业务服务、AI 能力、直播管控均在 Python 后端单服务内实现，无需跨服务、跨语言调用，本地运行延迟更低、资源占用更少、打包分发更简单，架构分层如下：

```Plain Text
┌─────────────────────────────────────────────────────────┐
│  用户层：Electron桌面端渲染进程                          │
│  对应功能：UI界面渲染、播放控制、互动操作、音频可视化    │
├─────────────────────────────────────────────────────────┤
│  Electron主进程（全局调度中枢）                          │
│  核心能力：窗口生命周期管理、Python后端服务启停、进程通信 │
├─────────────────────────────────────────────────────────┤
│  全Python一体化后端核心（单服务多模块架构）              │
│  ├─ API接口层：FastAPI路由、参数校验、响应格式化         │
│  ├─ 业务服务层：直播管控、歌单管理、用户体系、互动调度   │
│  ├─ AI能力模块：LLM文案生成、TTS语音合成、内容安全审核  │
│  ├─ 核心引擎模块：Socket.IO实时通信、音频混音处理、     │
│  │   直播调度引擎（APScheduler）、任务调度               │
│  ├─ 数据存储模块：本地数据库操作、文件资源管理            │
│  └─ 工具函数库：通用音频处理、时间格式化、配置读取        │
├─────────────────────────────────────────────────────────┤
│  本地资源层                                              │
│  存储：SQLite本地单文件数据库、本地音乐库、AI模型文件    │
│  工具：内置FFmpeg音频处理组件、Python运行环境            │
└─────────────────────────────────────────────────────────┘
```

**架构核心优势**：

- 全链路 Python 实现，业务逻辑与 AI 能力无缝集成，无跨语言调用开销，响应延迟大幅降低

- Python 后端打包为单可执行文件，无需部署多个独立服务，启动速度更快，稳定性更强，适配本地一键启动

- Python 生态丰富，音频处理、AI 模型、业务开发全场景覆盖，二次开发门槛更低

- 本地 AI 模式下完全本地化运行，无网络依赖，离线场景下所有功能均可正常使用（云 API 模式需网络连接）

### 3\.5 本地直播流核心方案（全 Python 原生实现）

采用**本地全时序同步直播方案**，基于 Python 异步调度能力实现，完全适配本地单机使用场景，无需公网带宽、无需额外直播服务，零配置启动：

1. 核心逻辑：Python 后端通过 APScheduler 维护全局唯一的直播时间轴、播放队列、播出计划，通过 Socket.IO 实时向前端同步时序状态，前端严格按照服务端时间轴校准播放进度，实现播放进度 100% 精准同步

2. 音频混音方案：基于 pydub\+ffmpeg\-python 实现 AI 主播 TTS 语音与歌曲音频的本地实时无缝混音，在歌曲间隙自动插入口播内容，无卡顿、无断层

3. 播放调度方案：Python 后端统一管控歌曲播放、口播插入、队列更新的全流程时序，前端仅负责音频渲染与用户交互，所有业务逻辑均在后端统一处理，保证直播逻辑的一致性

4. 核心优势：全 Python 链路实现，开发难度低、本地运行无网络依赖、时序控制精度毫秒级、播放延迟控制在 100ms 以内，可平滑扩展为局域网多设备同步收听

**后端分层架构说明**：

- `api/`（接口层）：基于 FastAPI Router 定义 HTTP 接口，负责请求路由、参数校验、响应格式化，不包含业务逻辑
- `service/`（业务服务层）：实现核心业务逻辑，处理数据编排与模块协调，被 api/ 层调用
- `core/`（核心引擎层）：提供底层能力（直播调度、音频处理、实时通信），被 service/ 层调用
- `db/`（数据存储层）：封装数据库模型与数据访问操作，被 service/ 层调用
- `ai/`（AI 能力层）：封装 LLM、TTS、内容安全等 AI 能力，被 service/ 层调用

调用关系：`api/ → service/ → core/ + ai/ + db/`

## 7 常见问题与解决方案

### 7\.1 软件启动后一直加载，无法进入主界面

- 核心原因：Python 后端服务启动失败、端口被占用、AI 模型文件缺失或损坏、权限不足

- 解决方案：

    1. 关闭占用 8765 端口的其他软件，确保端口可用

    2. 检查 AI 模型文件是否完整放置在指定目录，若缺失重新下载模型文件

    3. Windows 用户以管理员身份运行软件，Mac 用户在系统设置中放行应用权限

    4. 查看软件日志文件，定位具体启动失败原因，重新安装软件修复依赖缺失问题

### 7\.2 低配电脑运行卡顿，CPU / 内存占用过高

- 核心原因：本地 AI 模型推理占用大量系统资源，设备配置不足

- 解决方案：

    1. 在软件设置中切换为「云 API 模式」，无需本地运行 AI 模型，大幅降低系统资源占用

    2. 关闭其他占用系统资源的软件，释放内存与 CPU 资源

    3. 在设置中切换为「轻量化 AI 模式」，使用更小体积的 AI 模型，降低资源占用

    4. 限制软件的 CPU 核心使用数，避免占用过多系统资源

### 7\.3 歌曲播放无声音，AI 口播无音频

- 核心原因：系统音频设备异常、FFmpeg 组件缺失、音频文件损坏、软件音频输出配置错误

- 解决方案：

    1. 检查系统音频设备是否正常工作，系统音量是否开启，是否选择了正确的音频输出设备

    2. 重新安装软件，确保内置的 FFmpeg 组件完整无缺失

    3. 检查播放的音频文件是否损坏，更换其他音频文件测试

    4. 在软件设置中重新选择音频输出设备，重置音频播放配置

### 7\.4 本地音乐无法导入或识别

- 核心原因：音频格式不支持、文件路径包含特殊字符、音频文件损坏、权限不足

- 解决方案：

    1. 软件支持 MP3、WAV、FLAC、AAC 格式的音频文件，其他格式请先转换为支持的格式

    2. 将音频文件放置在纯英文、无特殊字符的文件路径中，避免中文、空格、特殊符号导致的识别失败

    3. 检查音频文件是否可正常播放，重新下载完整的音频文件

    4. 给软件开放对应文件夹的读取权限，确保可正常访问音频文件

### 7\.5 软件关闭后 Python 后端进程仍在运行

- 核心原因：应用异常退出，导致后端进程未被正常关闭

- 解决方案：

    1. 正常通过软件的退出按钮关闭应用，避免直接结束进程

    2. 若出现进程残留，可在任务管理器（Windows）/ 活动监视器（Mac）中找到对应 Python 进程，手动结束

    3. 升级到最新版本软件，已优化进程生命周期管理逻辑，避免进程残留问题
